# 新フェーズのオーケストレーションスキルを生成する

`phase1-next` / `phase1-5-next` と同じ「planner → implementer → implementation-reviewer → security-reviewer」の流れを、新しいフェーズ向けに複製・カスタマイズして配置する。

このスキルは **1 ファイルを書き出して終わり**。生成後に新スキルを実際に走らせるのはユーザーの判断。

## 重要な制約

- **スキル名（`{{PHASE_ID}}`）に `.` は使えない**: Claude Code は skill 名を `[a-z0-9-]+` に正規化する。ユーザー入力に `.` があったら **必ず `-` に置換する**。
  - 例: ユーザーが「Phase 1.5 用に作って」と言ったら `{{PHASE_ID}} = phase1-5`（`phase1.5` ではない）。
  - 一方、`{{PHASE_DISPLAY}}` や `{{PLAN_FILE}}` の `.` はそのまま残す。
- **テンプレートの構造は変えない**: 既存スキル（`phase1-next` / `phase1-5-next`）と同じ手順・ガードレールを保つこと。ユーザーが指示する追加要素（依存関係 / セキュリティスコープ等）は所定のプレースホルダにだけ流し込む。
- **既存スキルを上書きしない**: 出力先 `.claude/skills/{{PHASE_ID}}-next/SKILL.md` がすでに存在する場合は、ユーザーに上書き可否を確認してから書く。

## 手順

### Step 1: フェーズ識別子の決定

ユーザーが起動時にフェーズを指定していない場合は質問する。指定があれば確認だけ。

ユーザーから受け取った原文 `RAW`（例: `1.5`, `phase2`, `Phase 2`）を以下のとおり正規化する:

| 変数 | 値の決定方法 | 例（RAW=`1.5`） | 例（RAW=`phase2`） |
| --- | --- | --- | --- |
| `{{PHASE_DISPLAY}}` | `Phase ` + 数字部分（`.` 残す） | `Phase 1.5` | `Phase 2` |
| `{{PHASE_ID}}` | `phase` + 数字部分、**`.` を `-` に置換**、小文字 | `phase1-5` | `phase2` |
| `{{PLAN_FILE}}` | `docs/phase{数字部分}-plan.md`（`.` 残す） | `docs/phase1.5-plan.md` | `docs/phase2-plan.md` |

ユーザーに正規化結果（3 値）を提示し、**確認を取る**。`{{PLAN_FILE}}` のパスはユーザーが上書き可。

### Step 2: 計画ファイルからマイルストーン情報を抽出

1. `{{PLAN_FILE}}` が存在しなければ Read で確認。**存在しない場合はここで停止**し、ユーザーに「先に計画ファイルを作ってください」と伝える（このスキルは計画ファイルを生成しない）。
2. ファイル内の `### M{N}: {タイトル}` 形式のマイルストーン見出しを Grep で抽出する（pattern: `^### M[0-9.]+:`）。
3. 抽出結果から以下を導出:
   - `{{MILESTONE_RANGE}}`: 最小〜最大（例: `M0〜M8`, `M1〜M5`）
   - `{{FIRST_M}}`: 最小マイルストーン（例: `M1`）
   - `{{SECOND_M}}`: その次（例: `M2`）。`description` 末尾の「FIRST をやったら SECOND には自動で進まない」で使う
   - `{{MILESTONE_HEADINGS}}`: 例 `### M1:` 〜 `### M5:`（テキスト挿入用）
4. ユーザーに抽出結果を見せて確認。

### Step 3: フェーズ固有情報の聞き取り

以下 4 項目をユーザーに **1 メッセージで** 質問する。デフォルト値を提示し、ユーザーが省略した場合はそれを採用。

1. **`{{PHASE_OVERVIEW}}`** — 1〜2 段落で本フェーズのスコープ・前提・特徴。
   - デフォルト案: 計画ファイルの冒頭概要をそのまま流用していいか確認。
2. **`{{DEPENDENCY_RULES}}`** — マイルストーン間の依存関係の説明文（Step 1 内 / ガードレール内で使う）。
   - 例（Phase 1）: 「依存関係は MN → M(N+1) の順序のみ。M0 が未完了のまま M1 に進めない」
   - 例（Phase 1.5）: 「M1 / M2 は独立で並行可。M3 → M4 → M5 は依存関係あり。前提未達なら警告を出してユーザー確認」
   - 例（依存なし）: 「すべてのマイルストーンが独立で並行可」
3. **`{{IMPLEMENTER_PER_MILESTONE}}`** — 各マイルストーンの implementer 選択ガイド（haiku か sonnet か）の bullet list。
   - 計画ファイルの内容から推定して **draft を提示**し、ユーザーに修正させる。型システム / 多ファイル横断 / 複雑な型統合は sonnet、それ以外は haiku が原則。
4. **`{{SECURITY_SCOPE_RULES}}`** — どのマイルストーンで `security-reviewer` を呼ぶかのルール（段落 + bullet）。
   - 判断基準: 外部入力 / DB クエリ / 認証認可 / シークレット / CI/CD / 公開ワークフローのいずれかに触れるマイルストーン。
   - 計画ファイルの内容から draft を提示し、ユーザーに修正させる。
5. **`{{OUT_OF_SCOPE_NOTE}}`** — 本フェーズの対象外項目とその参照先。
   - 例: 「Phase 3 機能および受容リスクは本フェーズ対象外。`docs/backlog.md` §2 / §3 の管轄」
   - 例: 「次フェーズ機能を混入させないこと」

### Step 4: テンプレートを読んで置換

1. `.claude/skills/init-phase-skill/template.md` を Read で読む。
2. ファイル先頭のコメントブロック（`<!-- ... -->`）は **削除** して、frontmatter `---\nname: ...\n---` から始まるようにする。
3. すべての `{{PLACEHOLDER}}` を Step 1〜3 で確定した値で置換する。
4. 置換漏れがないか grep（Grep ツール）で `{{` を検索して確認。残っていたら停止してユーザーに確認。

### Step 5: 書き出し

1. 出力先: `.claude/skills/{{PHASE_ID}}-next/SKILL.md`
2. 出力先のディレクトリが存在しなければ Bash で `mkdir -p` する。
3. 出力先のファイルが既に存在する場合は **ユーザーに上書き確認**。
4. Write ツールで書き出す。
5. 書き出し後、ユーザーに以下を 1 メッセージで報告:
   - 生成パス
   - 主要な変数の確定値（`{{PHASE_DISPLAY}}` / `{{PHASE_ID}}` / `{{PLAN_FILE}}` / `{{MILESTONE_RANGE}}`）
   - 次のステップ案: 「Claude Code を再起動するか `/help` 等でスキル一覧を更新すれば `/{{PHASE_ID}}-next` が呼べるはず」

## ガードレール

- **template.md の構造を勝手に変えない**。プレースホルダ以外の本文（手順・ガードレール・引数説明）は既存スキルとの整合性を保つために固定する。テンプレ自体を直したいときは別タスクとして扱う。
- **`.` → `-` 置換を忘れない**: `{{PHASE_ID}}` では必ず置換、`{{PHASE_DISPLAY}}` / `{{PLAN_FILE}}` では残す。
- **計画ファイルの存在確認を飛ばさない**: 計画ファイルがないままスキルだけ作ると Step 1 で必ずコケる。Step 2 で停止する。
- **既存スキルを上書きするときは必ず確認**: ユーザーの手作業（`phase1-5-next` のような微調整）を消さないため。
- 途中でユーザーの意図が読み切れなかったら、勝手に補完せず質問する。
