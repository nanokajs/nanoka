# GitHub Issue を 1 件実装して閉じる

このスキルは **Issue 1 件を完了させたら止まる**。次の Issue に進むかはユーザーが判断する。

## フロー概要

```
Issue取得 → プラン策定（planner）→ ドラフトPR作成（sonnet、コメントのみ）
→ PRウォッチ → ユーザー承認 → 並列実装（haiku）→ 完了処理
```

---

## 手順

### Step 1: Issue の取得とブランチ準備

0. **中断セッションの復帰チェック**: 引数の Issue 番号が判明したら、まず `.claude/plan/plan-{番号}.md` が存在するか確認する。
   - 存在する場合: 前回セッションが中断されている。ファイルを Read してチェック済み・未チェックの状況をユーザーに提示し、「未完了のステップから再開しますか？」を確認する。Yes なら Step 4 から再開（Step 2・3 はスキップ）。
   - 存在しない場合: 通常通り Step 1 を続ける。
1. 引数で Issue 番号が指定されていない場合、ユーザーに番号を聞く。
2. `gh issue view {番号} --json number,title,body,labels,assignees,milestone,state` で Issue を取得する。
3. デフォルトブランチ（`main`）を最新にする:
   ```bash
   git checkout main && git pull origin main
   ```
4. Issue 番号をもとに作業ブランチを切る:
   ```bash
   git checkout -b issue-{番号}-{kebab-case-summary}
   ```
   ブランチ名の `{kebab-case-summary}` は Issue タイトルから 3〜5 語程度の英語 kebab-case で付ける。

### Step 2: Issue の確認

1. Issue の内容（タイトル・本文・ラベル・マイルストーン）をユーザーに見せ、**「この Issue を実装していいか」を確認**する。
   - `state: closed` の場合は警告を出してユーザー確認。
   - ユーザーが No と言ったらその理由を聞いて終了。
2. `docs/implementation-status.md` を Read して、Issue が既存の実装済み範囲・対象外範囲と衝突していないか確認する。衝突がある場合はユーザーに提示して判断を仰ぐ。

### Step 3: プラン策定（planner エージェント）

**ユーザー確認は行わない。** planner の結果が返ってきたら即 Step 3.5 へ進む。

1. `Agent` ツールで `subagent_type: "planner"` を呼び出す。プロンプトには以下を含める:
   - Issue 番号・タイトル・本文の全文
   - 「Issue のスコープだけで計画を作ること。関連しない改善や追加機能は含めないこと」を明示
   - `docs/nanoka.md` の load-bearing architectural rules を守ること
   - 「関連ファイルを先に Read してから計画を立てること」を明示
   - `packages/nanoka` または `packages/create-nanoka-app` に変更が及ぶ場合、その API・挙動変更を `docs-site/` 内の該当ドキュメントにも反映する作業をプランに含めること
2. planner の返答をユーザーに見せる（参考情報として）。ただし**承認を求めずそのまま次へ進む**。

### Step 3.5: スコープ評価とサブ Issue 分割

planner のプランが返ってきたら、**実装に着手する前に**スコープの大きさを評価する。

以下の基準のいずれかに該当する場合は「スコープが大きい」と判断する:

- 独立して動作する機能・変更が 2 つ以上含まれている
- 変更ファイルが 6 件以上
- planner が「フェーズ分け」「段階的に」「別途」などの言葉を使っている
- 1 つの PR にまとめるとレビューが困難な複雑さ

**スコープが大きい場合の手順:**

1. プランを分割単位（サブ Issue）に分ける。それぞれが独立してマージ可能な単位にする。
2. ユーザーに分割案を提示し、承認を取る。
3. 承認後、元の Issue にコメントを追加してトラッキングリストを記載:
   ```bash
   gh issue comment {元番号} --body "スコープが大きいため以下のサブ Issue に分割します:
   - [ ] #{サブ1番号} {タイトル1}
   - [ ] #{サブ2番号} {タイトル2}
   ..."
   ```
4. 各サブ Issue を作成（元 Issue を親として参照）:
   ```bash
   gh issue create --title "{サブタイトル}" --body "Parent: #{元番号}\n\n{内容}"
   ```
5. **最初のサブ Issue（#サブ1番号）だけを対象として以降のステップを続ける。**
   - 以降の Step では「元 Issue」ではなく「サブ Issue #サブ1番号」を処理対象とする。
   - 作業ブランチを切り直す（必要であれば）:
     ```bash
     git checkout -b issue-{サブ1番号}-{kebab-case-summary}
     ```
6. ユーザーに「サブ Issue #{サブ1番号} の実装に進みます」と伝えてから Step 3.6 へ。

**スコープが適切な場合:** 評価結果をユーザーに一言伝えてそのまま Step 3.6 へ。

### Step 3.6: プランファイルの作成

ユーザー承認済みのプランを `.claude/plan/plan-{番号}.md` としてチェックボックス形式で書き出す。

ファイルフォーマット:

```markdown
# Plan #42: {Issue タイトル}

Branch: {ブランチ名}

## 実装ステップ

- [ ] {ステップ1の説明}
- [ ] {ステップ2の説明}
- [ ] {ステップ3の説明}
...
```

- ステップは planner が返したプランの実装ステップを**1ステップ1行**で列挙する。粒度は「1ファイルへの変更」「1テストの追加」程度が目安。
- ファイルの書き出し後、パスをユーザーに一言伝えてから Step 4 へ。

### Step 4: ドラフト PR の作成（implementer-sonnet エージェント）

**目的: コードを書かず、実装意図を日本語コメントとして各ファイルの実装箇所に記述し、ドラフト PR を作る。**

1. `Agent` ツールで `subagent_type: "implementer-sonnet"` を呼ぶ。プロンプトには以下を含める:
   - 確定したプランの全文
   - プランファイルのパス: `.claude/plan/plan-{番号}.md`
   - Issue 番号・タイトル
   - **以下の指示を必ず含める**:
     > **コードを実装しないこと。** 代わりに、プランで変更が必要な各ファイルの実装箇所に、以下の形式で日本語の自然言語コメントだけを追加すること。
     >
     > ```
     > // TODO(#番号): {何を・なぜ・どう実装するかを日本語で記述}
     > // 例: ここに refreshToken フィールドを追加する。Cookie ヘッダーから取り出した値を
     > //     validateToken() に渡して検証後、セッションに保存する必要がある。
     > ```
     >
     > コメントは「何を」「なぜ」「どう実装するか」が後続の実装者に伝わる粒度で書くこと。
     > 全ファイルへのコメント追加が完了したら、以下の手順でドラフト PR を作成すること:
     > ```bash
     > git add <変更ファイル>
     > git commit -m "draft(#{番号}): add implementation intent comments"
     > git push -u origin {ブランチ名}
     > gh pr create --draft --title "draft: {タイトル} (#{番号})" \
     >   --body "$(cat <<'EOF'
     > ## 実装意図コメント
     >
     > このPRは実装意図を日本語コメントとして記述したドラフトです。
     > 承認後、実際のコード実装に移行します。
     >
     > Closes #{番号}
     > EOF
     > )"
     > ```
     > PR URL を報告すること。

2. implementer-sonnet から PR URL が返ってきたら、URL をユーザーに伝える。

### Step 5: PR のウォッチとユーザー承認待ち

1. 作成されたドラフト PR の URL をユーザーに提示する。
2. **以下のメッセージでユーザーに承認を求める:**
   ```
   ドラフトPR を作成しました: {PR URL}

   PR に追加した実装意図コメントを確認してください。
   問題なければ「承認」または「実装して」と伝えてください。
   修正が必要な場合は内容を教えてください。
   ```
3. ユーザーから修正要求が来た場合:
   - 修正内容を implementer-sonnet に伝えてコメントを更新させる。
   - 更新後、再度確認を求める。
4. **ユーザーから承認の合図（「承認」「実装して」「OK」「LGTM」など）が来るまでここで待機する。**

### Step 6: 並列実装（implementer エージェント）

ユーザーの承認を受けたら、プランのステップを**並列実行可能なグループ**に分けて haiku 実装者を複数呼ぶ。

#### 6-0: 並列グループの分割

プランのステップを依存関係で整理し、独立して実装できるグループに分ける。

- **グループ A**: 他に依存しないファイル群（型定義、ユーティリティ、テストなど）
- **グループ B**: グループ A に依存するファイル群
- **グループ C**: グループ B に依存するファイル群（順次実行）

依存がなければ全ステップを 1 グループにして 1 エージェントに渡す。

#### 6-1: 並列実装の実行

1. 独立したグループが複数ある場合、**1 メッセージで複数の `Agent` ツールを同時呼び出し**する（並列実行）。
   各エージェントのプロンプトには:
   - 担当するステップの一覧（プランから抜粋）
   - 「プランファイルの自分の担当ステップだけを実装すること」を明示
   - プランファイルのパス: `.claude/plan/plan-{番号}.md`
   - **以下の指示を含める**:
     > 実装前に `.claude/plan/plan-{番号}.md` を Read して自分の担当ステップを確認すること。
     > 各ステップ完了後、`- [ ]` を `- [x]` に更新してから次へ進むこと。
     > 実装箇所には `// TODO(#番号):` コメントがあるので、それを実際のコードに置き換えること。
     > プランに書かれた変更だけを行うこと。
2. 全グループの実装が返ってきたら、変更ファイル一覧をユーザーに見せる。
3. いずれかのエージェントがプランから逸脱した旨を報告した場合は、**そこで止めてユーザーに確認**。

#### 6-2: haiku 失敗時の sonnet 切り替え

haiku 版が以下の失敗パターンを示したら、該当グループだけ `implementer-sonnet` に切り替える:
- プランで指定された型を `any` にスタブして「完了」と報告した
- 既に行った修正を `git checkout` などで巻き戻した
- ツールエラーを「権限がロック」のような曖昧な言葉で諦めた
- 同じプランで 2 回続けてプラン未達を報告した

### Step 7: 実装レビュー（implementation-reviewer エージェント）

1. `Agent` ツールで `subagent_type: "implementation-reviewer"` を呼ぶ。プロンプトには:
   - Issue 番号・タイトル
   - 確定したプラン
   - implementer の完了報告（変更ファイル一覧）
   - 参照すべきドキュメント: `docs/nanoka.md` / `docs/implementation-status.md`
2. レビュー結果をユーザーに見せる。
3. **Critical / Major 指摘がある場合**: 修正のために implementer を再度呼ぶ。修正後、再レビュー。
4. Minor 指摘のみなら、ユーザーに「反映するか / 後回しにするか」を確認。

### Step 8: セキュリティレビュー（security-reviewer エージェント）

以下のいずれかに触れる変更がある場合のみ実行:
- HTTP body / params / query / headers の処理
- Zod バリデータや Hono バリデーション
- DB クエリ構築・Drizzle エスケープハッチ・D1/Turso アダプタ・バッチ動作
- OpenAPI 生成（バリデーション/強制のソースとして扱われる可能性がある場合）
- 認証・認可・セッション・トークン・API キー・シークレット
- CORS・リダイレクト・外部 fetch・ロギング・エラーメッセージ
- 新しい npm 依存関係

判断に迷ったら実行する。

1. `Agent` ツールで `subagent_type: "security-reviewer"` を呼ぶ。
2. Critical / Major 指摘があれば implementer を呼んで修正、再レビュー。
3. 結果をユーザーに見せる。

### Step 9: 完了処理

1. `.claude/plan/plan-{番号}.md` を Read して全ステップが `[x]` になっていることを確認する。未チェックが残っている場合は implementer を呼んで対応させる。全チェック確認後、ファイルを削除する:
   ```bash
   rm .claude/plan/plan-{番号}.md
   ```
2. `docs/implementation-status.md` に変更が反映すべき内容があれば更新する。
3. `packages/nanoka` または `packages/create-nanoka-app` に変更がある場合、`docs-site/` 内のドキュメントに API 変更が反映されているか確認し、未反映があれば更新する。
4. 完了確認コマンドがある場合（テスト・typecheck など）、ユーザーに「実行して回帰確認していいか」を聞く。
5. **ユーザーに以下のサマリを提示する**:
   ```
   ✅ #{Issue番号} {タイトル} 完了
   - 変更ファイル: ...
   - テスト: ...
   - docs 更新: ...
   ```
6. 変更をコミットして PR を ready for review に変更する:
   ```bash
   git add <変更ファイル>
   git commit -m "feat/fix/...: {タイトル} (#{番号})"
   git push
   gh pr ready {PR番号}
   ```
7. PR URL をユーザーに見せる。
8. **ここで停止**。Issue のクローズは PR マージ時に自動で行われる（`Closes #番号` による）。

## ガードレール

- **1 回の起動で 1 Issue だけ**。複数 Issue を跨いではいけない。
- **ユーザー確認ゲートを飛ばさない**:
  - Step 2 の Issue 着手確認
  - Step 5 の承認（コメント確認後に実装開始）
- **planner / implementer / reviewer の役割を混同しない**。このスキルを実行する Claude はオーケストレータ。コードを直接書かず、レビューも直接行わない。
- Step 4 の sonnet 実装者は**コードを書いてはいけない**。日本語コメントとドラフト PR 作成だけが役割。
- Step 6 で並列実行する際は**依存関係を必ず確認**する。型定義が未実装なのに利用側を先に実装しない。
- Issue のスコープを超えた改善・リファクタリングは提案に留め、実装しない。
- スコープが大きいと判断した場合、**勝手に全サブ Issue を実装しない**。最初の 1 件だけ進める。
- 途中でエラー・矛盾・スコープ膨張を検知したら、自動で続行せず**ユーザーに判断を仰ぐ**。
- `docs/nanoka.md` の load-bearing architectural rules に反する変更を検知したらブロックしてユーザーに確認。

## 引数

`/issue-draft {番号}` の形式で Issue 番号を直接渡せる。番号を省略した場合は Step 1 で質問する。

例:
- `/issue-draft 42` — Issue #42 を処理
- `/issue-draft` — Issue 番号を対話的に確認
