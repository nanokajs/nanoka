# 新フェーズの計画ドキュメントを生成する

Phase 2 / Phase 3 等の **計画ドキュメント本体** をスキャフォールドする。`docs/phase1-plan.md` / `docs/phase1.5-plan.md` と同じ章立ての skeleton を出力し、各マイルストーンの中身（具体的なチェックボックスタスクと完了基準）はユーザーが後で追記する前提。

このスキルは **1 ファイルを書き出して終わり**。出力後に `init-phase-skill` を呼んでオーケストレーションスキルを生やすところまでがワンセット。

## 重要な制約

- **計画ドキュメントのファイル名はドット (`.`) を残す**: `Phase 1.5` → `docs/phase1.5-plan.md`. ファイルシステムは `.` を許容するし、既存の `docs/phase1.5-plan.md` と命名規則を揃える。
- **既存計画ファイルを上書きしない**: 出力先がすでに存在する場合は、ユーザーに上書き可否を必ず確認。
- **マイルストーンの中身まで埋めない**: 本スキルはタイトルと skeleton までを生成する。各マイルストーンのサブタスクと完了基準はユーザーが追記する。スキル側で勝手に推測して埋めない。

## 手順

### Step 1: フェーズ識別子の決定

ユーザーが起動時にフェーズを指定していない場合は質問する。指定があれば確認だけ。

ユーザーから受け取った原文 `RAW`（例: `2`, `phase2`, `Phase 2`, `1.5`）を以下のとおり正規化:

| 変数 | 値の決定方法 | 例（RAW=`2`） | 例（RAW=`1.5`） |
| --- | --- | --- | --- |
| `{{PHASE_DISPLAY}}` | `Phase ` + 数字部分（`.` 残す） | `Phase 2` | `Phase 1.5` |
| 出力ファイルパス | `docs/phase{数字部分}-plan.md`（`.` 残す） | `docs/phase2-plan.md` | `docs/phase1.5-plan.md` |

ユーザーに正規化結果（表示名・出力先）を提示し、**確認を取る**。出力先パスはユーザーが上書き可。

### Step 2: 出力先の存在確認

1. 出力先ファイルを Read で読もうとする。**すでに存在する場合は内容の冒頭をユーザーに見せた上で上書き可否を確認**。No なら停止。
2. 出力先ディレクトリ（通常 `docs/`）が存在しなければ Bash で `mkdir -p` する。

### Step 3: 概要・ゴールの聞き取り

以下を **1 メッセージで** ユーザーに質問する:

1. **`{{PHASE_OVERVIEW}}`** — 1〜2 段落で本フェーズのスコープ・前提・モチベーション。
   - 例（Phase 1.5）: 「Phase 1（M0〜M8）の完了後、Phase 2 に着手する前に、公開ライブラリとしての運用基盤・型改善・onboarding 体験の穴を埋めるためのフェーズ。`docs/backlog.md` の §1 と §4 をマイルストーン化したもの。」
2. **`{{GOALS}}`** — 番号付き bullet で 2〜5 個のゴール。
   - 例:
     ```
     1. README どおりに第三者が `pnpm create hono@latest` から始めて Nanoka を導入できることが CI で保証される
     2. PR 単位で build/test/typecheck/lint が回り、tag push で publish できる CI/CD 基盤
     ```

### Step 4: マイルストーン構成の聞き取り

ユーザーに **マイルストーン一覧** を `M{N}: {タイトル}` の形式で 1 行ずつ提示してもらう。

例:
```
M1: 型・テストの持ち越し解消
M2: ドキュメント整備
M3: CI/CD 基盤
M4: Onboarding parity CI
M5: Publish 自動化
```

各マイルストーンの skeleton として、以下を生成する:

```markdown
### M{N}: {タイトル}

- [ ] TODO: サブタスクを記述
- [ ] TODO: サブタスクを記述

完了基準: TODO
```

複数マイルストーンを `\n\n` で連結して `{{MILESTONES}}` に流し込む。

**注意**: ユーザーがマイルストーンの中身（サブタスク）まで提供してきた場合のみ、それを反映する。聞かずに勝手に推測しない。

### Step 5: 依存関係・対象外の聞き取り

以下を **1 メッセージで** 質問する:

1. **`{{DEPENDENCIES}}`** — マイルストーン間の依存関係の説明文。
   - デフォルト案を提示: 「すべてのマイルストーンが独立で並行可。」
   - 例（Phase 1.5 風）: 「M1 / M2 は独立で並行可。M3 → M4 → M5 は依存関係あり。」
2. **`{{OUT_OF_SCOPE}}`** — 対象外項目とその参照先。
   - デフォルト案を提示: 「次フェーズ候補と受容リスクは本フェーズ対象外。`docs/backlog.md` §2 / §3 の管轄。」
   - 認証 / 複雑な query DSL のような全 Phase 外の項目があれば併記。

### Step 6: テンプレートを読んで置換

1. `.Codex/skills/init-phase-plan/template.md` を Read で読む。
2. ファイル先頭のコメントブロック（`<!-- ... -->`）は **削除** して、`# {{PHASE_DISPLAY}} 実装計画` から始まるようにする。
3. すべての `{{PLACEHOLDER}}` を Step 1〜5 で確定した値で置換する。
4. 置換漏れがないか Grep で `{{` を検索して確認。残っていたら停止してユーザーに確認。

### Step 7: 書き出し

1. 出力先（Step 1 で確定したパス、通常 `docs/phase{N}-plan.md`）に Write で書き出す。
2. ユーザーに以下を 1 メッセージで報告:
   - 生成パス
   - マイルストーン件数と一覧
   - 次のステップ案:
     - 各マイルストーンの skeleton にサブタスク（チェックボックス）と完了基準を追記してください
     - その後 `/init-phase-skill {phase-id}` でオーケストレーションスキルを生成できます

## ガードレール

- **template.md の構造を勝手に変えない**。プレースホルダ以外の章立て（## ゴール / ## マイルストーン / ## マイルストーンの依存関係 / ## 対象外 / ## 着手前チェック）は固定する。テンプレ自体を直したいときは別タスクとして扱う。
- **マイルストーンの中身を推測しない**: ユーザーが明示しない限り、サブタスクのチェックボックスは `TODO: サブタスクを記述` のままにする。
- **既存ファイルを必ず確認**: ユーザーの手作業を消さないため、Step 2 で必ず存在チェック。
- **ファイル名のドット (`.`) は残す**: `init-phase-skill` 側のスキル名（`phase1-5-next` など）と違い、計画ファイル名は `phase1.5-plan.md` のように `.` をそのまま使う。
- 途中でユーザーの意図が読み切れなかったら、勝手に補完せず質問する。

## `init-phase-skill` との関係

- 本スキル (`init-phase-plan`) → 計画ドキュメントを生成
- `init-phase-skill` → そのフェーズのオーケストレーションスキル（`/{{phase-id}}-next`）を生成

新フェーズ着手の標準フロー:
1. `/init-phase-plan {phase}` で計画 skeleton を作る
2. ユーザーが各マイルストーンのサブタスク・完了基準を埋める
3. `/init-phase-skill {phase}` でオーケストレーションスキルを生やす
4. `/{{phase-id}}-next` で M1 から着手
