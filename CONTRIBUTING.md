# Contributing to Nanoka

Nanoka is a public library published to npm. This guide covers local development, testing, and the release process.

## Repository structure

Nanoka is organized as a `pnpm` workspace:

- **packages/nanoka/** — The public library (`@nanokajs/core` on npm). This is what gets published.
  - `src/` — Implementation (field DSL, model definition, router, adapters, CLI, codegen)
  - `__tests__/` — Integration tests
  - `dist/` — Built ESM and TypeScript declarations (generated by tsup, not in repo)
- **examples/basic/** — A minimal Cloudflare Workers application demonstrating Nanoka in practice.
- **docs/** — Design and current status documentation
  - `docs/nanoka.md` — Design document covering the thesis, load-bearing rules, and API boundaries
  - `docs/implementation-status.md` — Current shipped / pending split, with links to tracked follow-ups
- **LICENSE** — MIT license (same file referenced in published package)

## Prerequisites

- **Node.js >=20** — per `packages/nanoka/package.json` `engines.node` field
- **pnpm 9.15.x** — pinned in `packageManager` field; we recommend `corepack enable` to use it automatically

## Development workflow

All commands are run from the repository root.

### Common tasks

```bash
# Install all workspace dependencies
pnpm install

# Build the library (tsup: ESM + .d.ts)
pnpm build

# Build library only
pnpm -C packages/nanoka build

# Run library tests (Workers + Node environments)
pnpm -C packages/nanoka test

# Run library tests in Workers environment only
pnpm -C packages/nanoka test:workers

# Run library tests in Node environment only
pnpm -C packages/nanoka test:node

# Type-check the library
pnpm -C packages/nanoka typecheck

# Type-check the example app
pnpm -C examples/basic typecheck

# Run example app E2E tests
pnpm -C examples/basic test

# Lint and format check with Biome
pnpm lint

# Auto-format with Biome
pnpm format
```

### Before opening a pull request

Make sure these all pass:

```bash
pnpm -C packages/nanoka test
pnpm -C packages/nanoka typecheck
pnpm -C examples/basic typecheck
pnpm -C examples/basic test
pnpm lint
```

These checks also run automatically on every pull request and on pushes to `main` via GitHub Actions (`.github/workflows/ci.yml`). PRs cannot be merged until CI is green.

## Release process (automated via tag push)

Releases are automated. Pushing a `v*` tag to GitHub triggers `.github/workflows/publish.yml`,
which builds, tests, type-checks, and publishes the package to npm using
**npm Trusted Publisher (OIDC)** — no `NPM_TOKEN` is stored in this repository.

### Step 1: Bump version on main

Edit `packages/nanoka/package.json` and increment the `version` field following [Semantic Versioning](https://semver.org/). Commit this change to main:

```bash
git commit -am "chore: bump version to X.Y.Z"
git push origin main
```

### Step 2: Optional dry-run

Trigger the **Publish dry-run** workflow from the GitHub Actions UI (`workflow_dispatch`)
to inspect the tarball contents before tagging. This does not authenticate with npm, so it can be run anytime.

### Step 3: Tag and push

```bash
git tag vX.Y.Z
git push origin main vX.Y.Z
```

The publish workflow runs automatically. Watch its progress under the **Actions** tab.

### Step 4: Verify

Confirm the new version appears on the npm package page. The provenance attestation
should also be visible on the npm package page (generated by `npm publish --provenance` in CI under the Trusted Publisher OIDC flow).

## One-time setup: npm Trusted Publisher registration

The first publish requires a maintainer to register a Trusted Publisher on npmjs.com.
This is a one-time human step.

1. Sign in to https://www.npmjs.com with an account that has publish access to the package.
2. Navigate to the package settings → **Trusted Publisher** → **Add**.
3. Enter the following fields:
   - **Publisher**: GitHub Actions
   - **Organization or user**: `nanokajs`
   - **Repository**: `nanoka`
   - **Workflow filename**: `publish.yml` (filename only, no path)
   - **Environment name**: leave blank
4. Save.

> ⚠️ The workflow filename is matched exactly. If you rename `.github/workflows/publish.yml`,
> you must re-register the Trusted Publisher with the new filename.

After registration, enabling **Require two-factor authentication and disallow tokens**
in the npm Account Settings is recommended (hardens the account against token leaks
since OIDC publishing no longer needs tokens at all).

## Rollback / failure recovery

### If the publish workflow fails

1. Inspect the failed run under **Actions** tab to identify the failing step.
2. Delete the tag locally and remotely, fix the issue, and re-tag:
   ```bash
   git push --delete origin vX.Y.Z
   git tag -d vX.Y.Z
   # ... fix the issue, commit if needed ...
   git tag vX.Y.Z
   git push origin main vX.Y.Z
   ```

### If a broken version was already published

- **Within 72 hours of publish**: `npm unpublish @nanokajs/core@X.Y.Z` removes the version
  from the registry. Use sparingly — unpublishing causes downstream cache invalidation.
- **After 72 hours**: `npm deprecate @nanokajs/core@X.Y.Z "reason"` marks the version
  as deprecated. The version remains installable but warns on `npm install`.
- Then bump to the next patch version, fix forward, and re-publish.

## About `prepublishOnly`

The `prepublishOnly` script in `packages/nanoka/package.json` is intentionally **written as direct pnpm commands**, not npm script wrappers:

```json
"prepublishOnly": "pnpm build && pnpm test && pnpm typecheck"
```

### Why pnpm direct commands?

In a pnpm workspace, calling `npm run build` from within the `prepublishOnly` hook can break workspace symlinks and peer dependency resolution. This leads to a dangerous situation: the pre-publish checks pass in isolation, but the published package doesn't work in the same way it was tested.

Using `pnpm` directly guarantees that the same dependency resolution applies locally and during publish, catching integration issues before release.

### Defense-in-depth

Implemented in Phase 1.5 / M5 — runs in CI as well as locally for defense-in-depth. The `prepublishOnly` script remains as a local safety check even though CI also runs equivalent checks. This protects against accidental local publishes.

## Notes for contributors

- **`docs/implementation-status.md` is the source of truth** for the shipped / pending split. Update it when shipped API surface, roadmap, or scope classification changes.
- **Deferred features and accepted risks** are tracked in [GitHub Issues](https://github.com/nanokajs/nanoka/issues). If you encounter a new deferral or risk, open or update an issue rather than adding a backlog file entry.
- **Deferred features** such as relations, typed query helpers, editor integrations, auth, full-stack React, and Drizzle replacement DSL remain outside the current core scope unless a GitHub Issue explicitly changes that scope.
- When you change `packages/nanoka/README.md` Install or Quickstart sections, also update `e2e/onboarding/fixture/` to match. The `Onboarding parity` CI workflow will fail otherwise.
