# Contributing to Nanoka

Nanoka is in Phase 1.5, building out the public library's operational foundation. This guide documents the development workflow and manual release process that contributors should follow.

## Repository structure

Nanoka is organized as a `pnpm` workspace:

- **packages/nanoka/** — The public library (`@nanokajs/core` on npm). This is what gets published.
  - `src/` — Implementation (field DSL, model definition, router, adapters, CLI, codegen)
  - `__tests__/` — Integration tests
  - `dist/` — Built ESM and TypeScript declarations (generated by tsup, not in repo)
- **examples/basic/** — A minimal Cloudflare Workers application demonstrating Nanoka in practice.
- **docs/** — Design and progress documentation
  - `docs/nanoka.md` — Design document covering the thesis, load-bearing rules, and API boundaries
  - `docs/phase1.5-plan.md` — Current phase progress and milestones (source of truth for what's next)
  - `docs/phase1-plan.md` — Completed Phase 1 history (archived, for reference)
  - `docs/backlog.md` — Deferred features (Phase 2 candidates), accepted risks, and operational notes
- **LICENSE** — MIT license (same file referenced in published package)

## Prerequisites

- **Node.js >=20** — per `packages/nanoka/package.json` `engines.node` field
- **pnpm 9.15.x** — pinned in `packageManager` field; we recommend `corepack enable` to use it automatically

## Development workflow

All commands are run from the repository root.

### Common tasks

```bash
# Install all workspace dependencies
pnpm install

# Build the library (tsup: ESM + .d.ts)
pnpm build

# Build library only
pnpm -C packages/nanoka build

# Run library tests (Workers + Node environments)
pnpm -C packages/nanoka test

# Run library tests in Workers environment only
pnpm -C packages/nanoka test:workers

# Run library tests in Node environment only
pnpm -C packages/nanoka test:node

# Type-check the library
pnpm -C packages/nanoka typecheck

# Type-check the example app
pnpm -C examples/basic typecheck

# Run example app E2E tests
pnpm -C examples/basic test

# Lint and format check with Biome
pnpm lint

# Auto-format with Biome
pnpm format
```

### Before opening a pull request

Make sure these all pass:

```bash
pnpm -C packages/nanoka test
pnpm -C packages/nanoka typecheck
pnpm -C examples/basic typecheck
pnpm -C examples/basic test
pnpm lint
```

These checks also run automatically on every pull request and on pushes to `main` via GitHub Actions (`.github/workflows/ci.yml`). PRs cannot be merged until CI is green.

## Release process (manual)

This process will be automated in a future Phase 1.5 milestone (M5). Until then, releases are manual.

### Step 1: Bump version

Edit `packages/nanoka/package.json` and increment the `version` field following [Semantic Versioning](https://semver.org/). Commit this change:

```bash
git commit -am "chore: bump version to X.Y.Z"
```

### Step 2: Local verification (recommended)

Dry-run the publish to inspect the tarball contents:

```bash
pnpm -C packages/nanoka publish --dry-run
```

Verify that `dist/index.js`, `dist/index.d.ts`, and other compiled outputs are present, and that `src/`, `__tests__/`, and config files are excluded.

### Step 3: Create a git tag and push

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

### Step 4: Publish

```bash
pnpm -C packages/nanoka publish
```

The `prepublishOnly` hook will run automatically:

```bash
pnpm build && pnpm test && pnpm typecheck
```

This ensures that the final published code passes all checks.

## About `prepublishOnly`

The `prepublishOnly` script in `packages/nanoka/package.json` is intentionally **written as direct pnpm commands**, not npm script wrappers:

```json
"prepublishOnly": "pnpm build && pnpm test && pnpm typecheck"
```

### Why pnpm direct commands?

In a pnpm workspace, calling `npm run build` from within the `prepublishOnly` hook can break workspace symlinks and peer dependency resolution. This leads to a dangerous situation: the pre-publish checks pass in isolation, but the published package doesn't work in the same way it was tested.

Using `pnpm` directly guarantees that the same dependency resolution applies locally and during publish, catching integration issues before release.

### Future note

When Phase 1.5 / M5 implements CI-driven publishing, this `prepublishOnly` script will remain as a local safety check, even though CI will also run equivalent checks. This defense-in-depth approach protects against accidental local publishes.

## Notes for contributors

- **`docs/phase1.5-plan.md` is the source of truth** for progress tracking. When you complete a task, update the corresponding checkbox immediately (don't batch updates to the end of your session).
- **Deferred features and accepted risks** are documented in `docs/backlog.md`. If you encounter a new deferral or risk, add it to the appropriate section.
- **Deferred features** (Phase 2 and beyond) are listed in `docs/backlog.md` §3. These are out of scope for Phase 1.5.
- When you change `packages/nanoka/README.md` Install or Quickstart sections, also update `e2e/onboarding/fixture/` to match. The `Onboarding parity` CI workflow will fail otherwise.
