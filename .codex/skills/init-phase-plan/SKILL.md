# 新フェーズの計画ドキュメントを生成する

Phase 2 / Phase 3 などの計画ドキュメント本体をスキャフォールドする。`docs/phase1-plan.md` / `docs/phase1.5-plan.md` と同じ章立ての skeleton を出力し、各マイルストーンの具体的なチェックボックスと完了基準は、ユーザーが明示した場合だけ反映する。

## 重要な制約

- 計画ドキュメントのファイル名はドット (`.`) を残す: `Phase 1.5` → `docs/phase1.5-plan.md`。
- 既存計画ファイルを上書きしない。出力先が存在する場合は、内容の冒頭を確認してからユーザーに上書き可否を聞く。
- マイルストーンの中身を推測しない。タイトルと skeleton までを生成し、ユーザーが内容を出した場合だけ反映する。

## 手順

### Step 1: フェーズ識別子の決定

ユーザーがフェーズを指定していない場合は質問する。指定があれば正規化結果を確認する。

- `{{PHASE_DISPLAY}}`: `Phase ` + 数字部分。例: `Phase 2`, `Phase 1.5`
- 出力ファイルパス: `docs/phase{数字部分}-plan.md`。例: `docs/phase2-plan.md`, `docs/phase1.5-plan.md`

表示名と出力先を提示し、確認を取る。出力先パスはユーザーが上書きできる。

### Step 2: 出力先の存在確認

1. 出力先ファイルの有無を確認する。存在する場合は冒頭を見せ、上書き可否を確認する。No なら停止。
2. 出力先ディレクトリがなければ作成する。

### Step 3: 概要・ゴールの聞き取り

以下を 1 メッセージで質問する。

- `{{PHASE_OVERVIEW}}`: 1〜2 段落で本フェーズのスコープ・前提・モチベーション。
- `{{GOALS}}`: 番号付き bullet で 2〜5 個のゴール。

### Step 4: マイルストーン構成の聞き取り

ユーザーにマイルストーン一覧を `M{N}: {タイトル}` の形式で 1 行ずつ提示してもらう。

各マイルストーンの skeleton は次の形にする。

```markdown
### M{N}: {タイトル}

- [ ] TODO: サブタスクを記述
- [ ] TODO: サブタスクを記述

完了基準: TODO
```

ユーザーがサブタスクや完了基準まで提供した場合のみ、それを反映する。

### Step 5: 依存関係・対象外の聞き取り

以下を 1 メッセージで質問する。

- `{{DEPENDENCIES}}`: マイルストーン間の依存関係。デフォルト案は「すべてのマイルストーンが独立で並行可。」
- `{{OUT_OF_SCOPE}}`: 対象外項目と参照先。例: 次フェーズ候補や受容リスクは `docs/backlog.md` の管轄。

### Step 6: テンプレートを読んで置換

1. `.codex/skills/init-phase-plan/template.md` を読む。
2. 先頭コメントブロックは削除し、`# {{PHASE_DISPLAY}} 実装計画` から始まるようにする。
3. すべての `{{PLACEHOLDER}}` を確定値で置換する。
4. `rg '{{'` で置換漏れを確認する。残っていたら停止して確認する。

### Step 7: 書き出し

出力先へ書き出し、生成パス、マイルストーン件数と一覧、次のステップを報告する。

## ガードレール

- `template.md` の章立ては勝手に変えない。
- マイルストーンの中身を推測しない。
- 既存ファイルを消さない。
- 計画ファイル名のドット (`.`) は残す。
- 意図が読み切れない場合は補完せず質問する。

## `init-phase-skill` との関係

標準フロー:

1. `init-phase-plan {phase}` で計画 skeleton を作る。
2. ユーザーが各マイルストーンのサブタスク・完了基準を埋める。
3. `init-phase-skill {phase}` でオーケストレーションスキルを生成する。
4. `{phase-id}-next` で M1 から着手する。
